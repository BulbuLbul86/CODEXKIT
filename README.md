# CODEXKIT

CODEXKIT переносит рабочую Windows-среду между компьютерами с минимальным количеством ручных действий.

Главная идея: человек запускает скрипт впервые, заранее ничего не настраивает, а CODEXKIT сам анализирует локальную среду и собирает переносимый комплект.

## Что умеет

- находит и сохраняет рабочие настройки пользователя;
- собирает локальные git-репозитории вместе с незапушенными изменениями;
- сохраняет состояние редакторов, терминала, оболочек, пакетных менеджеров и других типовых инструментов, если они есть на компьютере;
- пишет отчет `environment-inventory.json`, где видно, что именно найдено;
- строит локальный индекс чатов, outputs-артефактов и состояния проектов;
- при повторном запуске обновляет уже собранный комплект инкрементально: добавляет новое и перезаписывает изменившееся;
- восстанавливает найденные настройки на другом Windows-компьютере;
- не требует 7-Zip, WinRAR или запуска от администратора для обычного сценария;
- позволяет добавить редкие личные пути через `custom-paths.json`.

## Быстрый старт

1. На текущем компьютере запусти `1-BEFORE-MOVE.bat`.
2. Перенеси папку `CODEXKIT` целиком на другой компьютер.
3. На новом компьютере запусти `2-RESTORE-HERE.bat`.

Готово.

Если внутри комплекта появился `codexkit-transfer.zip` или папка `codexkit-transfer-parts`, открывать их руками не нужно. Второй батник сам распакует переносимый архив штатными средствами Windows и сразу продолжит восстановление.

## Где должны лежать батники

Оба bat-файла должны лежать в корне папки `CODEXKIT`, рядом с PowerShell-скриптами:

```text
CODEXKIT\
  1-BEFORE-MOVE.bat
  2-RESTORE-HERE.bat
  refresh-codexkit.ps1
  restore-codexkit.ps1
  verify-codexkit.ps1
  README.md
```

Не вытаскивай bat-файлы отдельно из папки. CODEXKIT работает относительно своей папки, поэтому ее можно положить куда угодно: на флешку, внешний диск, рабочий стол или в загрузки.

После запуска первого bat-файла в этой же папке появятся рабочие данные: `state`, `repo-snapshots`, `codexkit-transfer.zip` или `codexkit-transfer-parts`. На новом компьютере все равно запускается только `2-RESTORE-HERE.bat` из корня этой же папки.

## Повторный запуск

`1-BEFORE-MOVE.bat` запускает сохранение в режиме `Auto`.

- если комплект собирается впервые, CODEXKIT делает полный снимок;
- если `state` или `repo-snapshots` уже есть, CODEXKIT обновляет их инкрементально: неизмененные файлы не копируются заново, новые и измененные добавляются поверх старого комплекта;
- архив `codexkit-transfer.zip` или папка `codexkit-transfer-parts` пересобираются из актуального содержимого комплекта, чтобы второй bat-файл восстанавливал свежую версию.

Если нужно именно очистить старую копию и собрать все заново, запусти из консоли:

```bat
set CODEXKIT_REFRESH_MODE=Full
1-BEFORE-MOVE.bat
```

Обычному сценарию это не нужно: достаточно снова запустить `1-BEFORE-MOVE.bat`.

## Как работает автоанализ

При сохранении `refresh-codexkit.ps1` проверяет типовые места, где разработчики и активные пользователи хранят рабочую среду:

- пользовательский профиль;
- настройки редакторов;
- настройки терминала и PowerShell;
- конфиги пакетных менеджеров;
- локальные ключи и CLI-настройки;
- типовые папки с проектами.

Скрипт копирует только то, что реально найдено на компьютере. Если файла или папки нет, он просто пропускает этот пункт.

## Проекты

Если `repo-manifest.json` отсутствует или пустой, CODEXKIT сам ищет git-репозитории в типовых рабочих папках:

- `Documents`
- `Documents\Codex`
- `source`
- `projects`
- `dev`
- `Desktop`
- дополнительные пути из `custom-paths.json`

На других дисках CODEXKIT также проверяет типовые папки вроде `D:\Projects`, `D:\dev`, `D:\source`, `D:\repos` и ищет свободно лежащие git-репозитории на ограниченной глубине. Это помогает найти проекты, которые лежат не в профиле пользователя, а, например, в отдельных папках на диске `D:`.

Старые переносимые комплекты `CODEXKIT`, `travel-kit`, их `repo-snapshots` и архивные части пропускаются при поиске, чтобы новый архив не начинал упаковывать предыдущие бэкапы.

Найденные репозитории попадают в `repo-snapshots/`. Это помогает перенести даже локальные изменения, которые еще не были отправлены в GitHub.

При снимке репозитория CODEXKIT исключает типовые кэши, результаты сборки и тяжелые данные: `build`, `dist`, `out`, `output`, `outputs`, `target`, `models`, `checkpoints`, `weights`, `datasets`, `data`, `cache`, `node_modules`, `.gradle`, `.cxx`, а также крупные бинарные артефакты вроде `*.img`, `*.iso`, `*.apk`, `*.zip`, `*.rar`, `*.safetensors`, `*.ckpt`, `*.pt`, `*.pth`, `*.onnx`. Если такой файл действительно нужен для переезда, добавь его явно через `custom-paths.json`.

## Отчет

После сохранения появляется `environment-inventory.json`.

В нем видно:

- какие файлы и папки найдены;
- какие элементы скопированы;
- какие ожидаемые элементы отсутствовали;
- где искались репозитории;
- сколько репозиториев найдено.

Также создается локальная карта работ:

- `docs\codex-work-index.md` — человекочитаемый список последних чатов, рабочих папок, outputs-файлов и репозиториев;
- `docs\codex-work-index.json` — машинный индекс для быстрого поиска и будущей автоматизации.

В индекс не копируются полные тексты чатов. Он хранит названия, даты, пути, размеры, статусы репозиториев и ссылки на локальные файлы, чтобы снизить риск случайного раскрытия секретов.

## Дополнительные пути

Обычно `custom-paths.json` не нужен. Используй его, если у тебя есть нестандартные секреты, сертификаты, сборки, конфиги или папки с проектами.

Пример:

```json
{
  "files": [
    {
      "category": "custom-vpn",
      "source": "%USERPROFILE%\\Secrets\\my-vpn.conf",
      "destination": "vpn\\my-vpn.conf",
      "target": "state"
    }
  ],
  "directories": [
    {
      "category": "custom-certs",
      "source": "%USERPROFILE%\\Certificates",
      "destination": "certs",
      "target": "state"
    }
  ],
  "repo_roots": [
    "%USERPROFILE%\\source",
    "D:\\Projects"
  ]
}
```

## Важные ограничения

- Сейчас пакет рассчитан на Windows.
- Внутри переносимого комплекта могут быть чувствительные данные: ключи, токены, конфиги, локальные базы и история инструментов.
- Не публикуй папки `state/`, `repo-snapshots/`, `docs/`, архивы и личный `custom-paths.json`.
- Если один архив получается слишком большим или рискованным для встроенного ZIP, CODEXKIT автоматически делает папку частей `codexkit-transfer-parts`. Каждая часть держится ниже лимита FAT32, а `2-RESTORE-HERE.bat` собирает комплект обратно без 7-Zip, WinRAR и ручной распаковки.
- Символические ссылки и junction-папки пропускаются при файловом копировании, чтобы комплект собирался без прав администратора и не уходил в системные петли. Для git-проектов сами репозитории и локальные изменения сохраняются обычными файлами.

## Что можно публиковать

Публично выкладывай только чистый шаблон:

- скрипты;
- `README.md`;
- пустой `custom-paths.json`;
- `bootstrap-packages.json`;
- `LICENSE`.

Не выкладывай рабочий комплект после запуска на своем компьютере.
