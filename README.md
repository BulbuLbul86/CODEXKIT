# CODEXKIT

CODEXKIT переносит рабочую Windows-среду между компьютерами с минимальным количеством ручных действий.

Главная идея: человек запускает скрипт впервые, заранее ничего не настраивает, а CODEXKIT сам анализирует локальную среду и собирает переносимый комплект.

## Что умеет

- находит и сохраняет рабочие настройки пользователя;
- собирает локальные git-репозитории вместе с незапушенными изменениями;
- сохраняет состояние редакторов, терминала, оболочек, пакетных менеджеров и других типовых инструментов, если они есть на компьютере;
- пишет отчет `environment-inventory.json`, где видно, что именно найдено;
- восстанавливает найденные настройки на другом Windows-компьютере;
- позволяет добавить редкие личные пути через `custom-paths.json`.

## Быстрый старт

1. На текущем компьютере запусти `1-BEFORE-MOVE.bat`.
2. Перенеси всю папку `CODEXKIT` на другой компьютер.
3. На новом компьютере запусти `2-RESTORE-HERE.bat`.

Готово.

## Как работает автоанализ

При сохранении `refresh-codexkit.ps1` проверяет типовые места, где разработчики и активные пользователи хранят рабочую среду:

- пользовательский профиль;
- настройки редакторов;
- настройки терминала и PowerShell;
- конфиги пакетных менеджеров;
- локальные ключи и CLI-настройки;
- типовые папки с проектами.

Скрипт копирует только то, что реально найдено на компьютере. Если файла или папки нет, он просто пропускает этот пункт.

## Проекты

Если `repo-manifest.json` пустой, CODEXKIT сам ищет git-репозитории в типовых рабочих папках:

- `Documents`
- `Documents\Codex`
- `source`
- `projects`
- `dev`
- `Desktop`
- дополнительные пути из `custom-paths.json`

Найденные репозитории попадают в `repo-snapshots/`. Это помогает перенести даже локальные изменения, которые еще не были отправлены в GitHub.

## Отчет

После сохранения появляется `environment-inventory.json`.

В нем видно:

- какие файлы и папки найдены;
- какие элементы скопированы;
- какие ожидаемые элементы отсутствовали;
- где искались репозитории;
- сколько репозиториев найдено.

## Дополнительные пути

Обычно `custom-paths.json` не нужен. Используй его, если у тебя есть нестандартные секреты, сертификаты, сборки, конфиги или папки с проектами.

Пример:

```json
{
  "files": [
    {
      "category": "custom-vpn",
      "source": "%USERPROFILE%\\Secrets\\my-vpn.conf",
      "destination": "vpn\\my-vpn.conf",
      "target": "state"
    }
  ],
  "directories": [
    {
      "category": "custom-certs",
      "source": "%USERPROFILE%\\Certificates",
      "destination": "certs",
      "target": "state"
    }
  ],
  "repo_roots": [
    "%USERPROFILE%\\source",
    "D:\\Projects"
  ]
}
```

## Важные ограничения

- Сейчас пакет рассчитан на Windows.
- Внутри переносимого комплекта могут быть чувствительные данные: ключи, токены, конфиги, локальные базы и история инструментов.
- Не публикуй папки `state/`, `repo-snapshots/`, архивы и личный `custom-paths.json`.
- Если носитель в `FAT32`, большой `codexkit-transfer.zip` может не создаться из-за лимита 4 ГБ. В этом случае переноси всю папку `CODEXKIT`.

## Что можно публиковать

Публично выкладывай только чистый шаблон:

- скрипты;
- `README.md`;
- пустой `custom-paths.json`;
- пустой `repo-manifest.json`;
- пустые служебные списки.

Не выкладывай рабочий комплект после запуска на своем компьютере.
